# Round 2 SEO Audit

日期：2026-04-09

## 审计范围

- 首页 `/`
- 项目详情页 `/projects/[id]`
- 语言聚合页 `/languages/[slug]`
- 主题聚合页 `/topics/[slug]`
- sitemap / robots
- 首页搜索与筛选结果页

## 本轮已落地修复

### 1. 搜索和筛选结果页不再争抢收录

- 首页现在会根据 `q` / `lang` / `page` 动态输出 metadata。
- 带搜索词、语言筛选或分页参数的首页结果页统一 `noindex, follow`。
- 首页 canonical 固定指向根路径 `/`。

价值：

- 避免搜索结果页、筛选页、分页页被重复收录。
- 减少“同一批项目列表被多个 URL 版本收录”的问题。

### 2. 详情页只有内容足够时才允许索引

- 项目详情页 metadata 现在会根据 `intro_status` / `wiki_status` / `seo_title` 判断是否具备索引价值。
- 没有生成出有效内容的详情页自动 `noindex, follow`。

价值：

- 避免大量薄内容详情页进入索引。
- 提高已生成内容项目页的整体质量密度。

### 3. sitemap 只提交有内容价值的项目页

- sitemap 不再提交所有项目页。
- 当前仅提交具备以下任一条件的项目：
  - `seo_title` 存在
  - `intro_status = completed`
  - `wiki_status = completed`

价值：

- 搜索引擎爬虫优先看到更有价值的详情页。
- 减少站内低质量 URL 对整体收录质量的稀释。

### 4. 结构化数据更完整

- 首页保留 `WebSite + SearchAction`
- 首页新增 `ItemList`
- 项目详情页保留 `SoftwareSourceCode`
- 项目详情页新增 `BreadcrumbList`
- 项目详情页在 FAQ 存在时继续输出 `FAQPage`
- 聚合页继续使用 `CollectionPage + BreadcrumbList + ItemList + FAQPage`

价值：

- 搜索引擎更容易理解页面角色、列表关系和详情页层级。
- 详情页的语义信号更完整。

### 5. 内链策略更清晰

- 详情页新增“相关入口”区块：
  - 当前语言页
  - 前 3 个 topic 聚合页
  - 全部主题入口
- 原有 related topics / related projects 保留
- 首页保留语言入口和主题入口区块

价值：

- 提升站内跳转深度
- 强化“详情页 -> 聚合页 -> 更多详情页”的 SEO 内链闭环

## 当前仍存在的 SEO 风险

### 高优先级

1. 详情页标题仍较依赖 LLM 输出质量

- 如果 `seo_title` 质量一般，title 仍可能偏模板化。
- 建议后续增加长度校验、去重策略和关键词覆盖规则。

2. 首页仍主要依赖项目卡片承接内容

- 首页已经具备入口价值，但“解释性正文”仍偏少。
- 如果未来想争取更强首页收录词，可以考虑增加一段稳定的说明模块。

### 中优先级

3. 聚合页排序策略偏通用

- 当前聚合页主要按现有项目集合输出，排序不够“意图化”。
- 后续可按 `stars + 内容完整度 + project_type` 调整排序。

4. 专题页仍主要是 language/topic 两类

- 这轮还没增加“用途型专题页”。
- 如果下一轮想进一步做 SEO 增长，建议优先补“自动专题页 2.0”。

5. FAQ 内容仍依赖首轮生成质量

- FAQ 已纳入结构化数据，但仍可能偏保守。
- 后续可增加 FAQ 最短长度、问题去重和意图覆盖校验。

## 下一轮建议

1. 增加用途型自动专题页
2. 为详情页 title / description 增加质量规则校验
3. 为聚合页增加更稳定的排序策略
4. 增加后台 SEO 质量检查面板
5. 增加 canonical / robots 的页面级测试

## 本轮结论

本轮 SEO 优化的重点不是“加更多页面”，而是先把收录边界收紧：

- 不让薄内容页乱进索引
- 只把有价值的详情页交给 sitemap
- 让首页、详情页、聚合页的结构化数据更清晰
- 让内链更自然地把权重传给聚合页和详情页

这一步完成后，后续再做“自动专题页 2.0”会更稳。
